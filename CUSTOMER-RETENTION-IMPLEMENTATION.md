# Customer Retention — Phase 1 Implementation

> Implementation record for the 7-PR Customer Retention initiative that introduces a durable store-issued customer code, multi-HOT-account management, returning-customer prefill, and identity-linked commissions.
>
> **Status:** PR1–PR7 complete. Staging-ready.
> **Scope:** Phase 1 only. Phase 2 items (phone correction, merge workflow, `/account` page, marketing) are out of scope — see [Known Gaps](#known-gaps).

---

## 1. Overview

### 1.1 What the feature delivers
- Every customer receives a durable store-issued code of the form `CLAL-XXXXXX`, generated by the database and immutable once assigned.
- Customers can own multiple HOT Mobile accounts through a new `customer_hot_accounts` table with 1:N semantics and soft-close transfers.
- Returning customers in the cart can restore their profile via the existing OTP flow (`/store/auth?return=/store/cart`) without re-typing their details.
- CRM admins manage HOT accounts and edit customer profile fields from the existing customers modal. All admin changes write to `audit_log`.
- CRM search accepts `customer_code` or `hot_mobile_id` lookups, and the orders page shows the customer code read-only.
- `commission_sales` rows are linked to `customer_id` and optionally `customer_hot_account_id` with a tracked `match_status` (matched / ambiguous / unmatched / manual / pending / conflict).

### 1.2 Guiding principles (enforced by every PR)
1. **Backward compatible.** No existing public API signature was broken. Order, auth, profile, and notifications endpoints continue to accept the same inputs.
2. **Database-generated identifiers.** `customer_code` is issued by a sequence-backed trigger — no application retry loops, no collision possible.
3. **OTP before prefill.** The cart never reveals customer data from a phone number alone. Prefill only happens after the customer returns from `/store/auth` with a valid bearer token.
4. **`id_number` stays admin-only.** It is not included in any public GET response and is not auto-filled for returning customers.
5. **Soft closes for HOT.** HOT accounts are never hard-deleted; the `cancel` action sets `status='cancelled'` + `ended_at`.
6. **Transfers via close + recreate.** Moving a HOT line between customers is done by closing the old row (`status='transferred'`) and inserting a new one, preserving the historical record.
7. **Reuse existing infrastructure.** We lean on `requireAdmin`, `validateBody`, `audit_log`, and `lib/customer-auth.ts` — no parallel helper stacks.

## 2. Schema changes

### 2.1 `customers` — new columns & triggers
Migration: `supabase/migrations/20260411000001_customer_retention_phase1.sql`

| Change | Type | Notes |
|---|---|---|
| `customer_code TEXT` | column | Unique when not null. Assigned by trigger on INSERT, protected on UPDATE. |
| `customer_code_seq` | sequence | `START 100000`, `MAX 999999`, `NO CYCLE`. |
| `assign_customer_code()` | trigger | `BEFORE INSERT` — sets `CLAL-######` if null. |
| `protect_customer_code()` | trigger | `BEFORE UPDATE` — raises if existing code is changed. Allows `NULL → code` for backfill. |
| `normalize_customer_phone()` | trigger | `BEFORE INSERT OR UPDATE OF phone` — strips spaces, dashes, dots. |
| `assign_code_to_existing_customer(UUID)` | RPC | `SECURITY DEFINER` trickle-in backfill used by the order route for legacy customers. |
| `idx_customers_customer_code` | partial unique index | `WHERE customer_code IS NOT NULL`. |
| `idx_customers_phone` | index | Safety net for phone lookups. |

### 2.2 `customer_hot_accounts` — new table
Created in the same migration.

```
id                  UUID PK (gen_random_uuid)
customer_id         UUID FK customers(id) ON DELETE CASCADE
hot_mobile_id       TEXT NOT NULL
hot_customer_code   TEXT
line_phone          TEXT
label               TEXT
status              TEXT CHECK IN ('pending','active','inactive','cancelled','transferred') DEFAULT 'active'
is_primary          BOOLEAN DEFAULT FALSE
source              TEXT CHECK IN ('admin_manual','sales_doc','order_sync','import') DEFAULT 'admin_manual'
source_order_id     TEXT
verified_at         TIMESTAMPTZ
verified_by_id      UUID
verified_by_name    TEXT
ended_at            TIMESTAMPTZ
notes               TEXT
created_by_id       UUID
created_by_name     TEXT
created_at          TIMESTAMPTZ DEFAULT NOW()
updated_at          TIMESTAMPTZ DEFAULT NOW() (updated by trigger)
```

Indexes and constraints:

| Index | Condition | Purpose |
|---|---|---|
| `idx_customer_hot_accounts_active_hot_id` | `UNIQUE (hot_mobile_id) WHERE status IN ('pending','active')` | Prevents two active customers from claiming the same HOT ID while still permitting transfer via close + recreate. |
| `idx_customer_hot_accounts_one_primary` | `UNIQUE (customer_id) WHERE is_primary AND status IN ('pending','active')` | Enforces a single active primary account per customer. |
| `idx_customer_hot_accounts_customer` | `(customer_id)` | Lookup by customer. |
| `idx_customer_hot_accounts_status` | partial on active/pending | Fast filtering in CRM. |

RLS is enabled; the only policy is `service_role_all`, so the table is only reachable from server routes running with the service key.

### 2.3 `commission_sales` — identity linking
Migration: `supabase/migrations/20260411000002_commissions_identity.sql`

| Column | Type | Purpose |
|---|---|---|
| `customer_id` | UUID FK customers ON DELETE SET NULL | Resolved customer for the sale. |
| `customer_hot_account_id` | UUID FK customer_hot_accounts ON DELETE SET NULL | Specific HOT line if known. |
| `hot_mobile_id_snapshot` | TEXT | Original HOT mobile id from the source (preserved even if HOT account is closed). |
| `store_customer_code_snapshot` | TEXT | Snapshot of the customer code at sale time (survives code backfill reorderings). |
| `match_status` | TEXT CHECK | `pending / matched / ambiguous / unmatched / conflict / manual`. |
| `match_method` | TEXT | `hot_mobile_id / phone / order_sync / manual`. |
| `match_confidence` | NUMERIC(3,2) | `1.0` for HOT / order sync, `0.9` for phone fallback. |

Indexes:
- `idx_commission_sales_customer` (partial, `WHERE customer_id IS NOT NULL`)
- `idx_commission_sales_hot_account` (partial)
- `idx_commission_sales_match_status`

## 3. API endpoints

### 3.1 New routes
| Method | Path | Auth | Purpose |
|---|---|---|---|
| `GET` | `/api/crm/customers/[id]` | Admin | Returns the customer row and its active/pending/inactive HOT accounts. |
| `PATCH` | `/api/crm/customers/[id]` | Admin | Whitelist update for `name, email, city, address, id_number, notes, segment, tags, birthday`. Writes to `audit_log`. |
| `POST` | `/api/crm/customers/[id]/hot-accounts` | Admin | Adds a HOT account. Enforces primary uniqueness, returns 409 on duplicate active `hot_mobile_id`. |
| `PATCH` | `/api/crm/customers/[id]/hot-accounts/[accountId]` | Admin | Updates editable fields. Setting `is_primary=true` automatically demotes siblings. Closing the account sets `ended_at`. |
| `DELETE` | `/api/crm/customers/[id]/hot-accounts/[accountId]` | Admin | Soft-close: sets `status='cancelled'`, `is_primary=false`, `ended_at=now()`. |

### 3.2 Changed routes
| Path | Change |
|---|---|
| `POST /api/orders` | Reads `customer_code` on existing customer, triggers trickle-in backfill for legacy nulls, returns `customerCode` + `isNewCustomer` in the response payload. Handles 23505 phone race by re-selecting. |
| `POST /api/auth/customer` (`verify_otp`) | Selects `customer_code` in both branches (existing + newly created) and returns it inside `customer` on the response. Does **not** return `id_number`. |
| `GET /api/customer/profile` | `lib/customer-auth.ts` helper now selects `customer_code`. Handler signature unchanged. |
| `GET /api/crm/customers` | Accepts `hot_mobile_id` query param. `customer_code` is included in the search ILIKE OR clause. |
| `POST /api/admin/commissions/sales` | Runs `matchSaleIdentity()` (HOT → phone → unmatched), populates all 7 new identity columns. |

### 3.3 Notifications + emails
`lib/bot/notifications.ts::notifyNewOrder` and `lib/email-templates.ts::orderConfirmationEmail` now accept an optional `customerCode` parameter. When the order is a **new** customer, the code is rendered into the WhatsApp template and an HTML card in the confirmation email. Returning customers do not see the "new code" block.

## 4. Store-side changes

### 4.1 Cart (`app/store/cart/page.tsx`)
- Adds a "Returning customer?" banner in the delivery-info step. The banner has a single CTA that redirects to `/store/auth?return=/store/cart`.
- On mount, reads the existing `localStorage.clal_customer_token`. If present, calls `GET /api/customer/profile`, reuses the bearer, and prefills `name, phone, email, city, address`.
- When the customer already typed data before returning from auth, a `confirm()` warns before overwriting.
- `id_number` is intentionally never auto-filled. The customer re-types it — security over ergonomics (see [Architectural Decisions](#architectural-decisions)).
- Stale tokens (401) are purged with `localStorage.removeItem` for both `clal_customer_token` and `clal_customer`, so the cart falls back to the blank flow without user interruption.

### 4.2 Confirm step (`components/store/cart/ConfirmStep.tsx`)
New block renders the customer code for new customers only (`order.customerCode && order.isNewCustomer`):
- Glass card with centered `CLAL-XXXXXX` in brand color, wide letter spacing.
- Tagline: "احتفظ به لطلباتك القادمة".

### 4.3 Auth page (`/store/auth`) — unchanged
The existing `?return=` redirect and `localStorage.clal_customer_token` writes are reused verbatim. Switching to `sessionStorage` would break `/store/account`, so this was explicitly held out of scope.

## 6. Commissions linking

### 6.1 Auto-sync (`lib/commissions/sync-orders.ts`)
- Orders query now selects `assigned_to, customer_id` alongside the financial fields.
- Two preload maps are built up front:
  - `customerCodeMap`: `customer_id → customer_code`, loaded from `customers`.
  - `primaryHotMap`: `customer_id → { id, hot_mobile_id }`, loaded from `customer_hot_accounts` where `is_primary AND status IN ('pending','active')`.
- Each inserted `commission_sales` row is populated with `customer_id`, `customer_hot_account_id`, `hot_mobile_id_snapshot`, `store_customer_code_snapshot`, `match_status='matched'`, `match_method='order_sync'`, and `match_confidence=1.0`. Orders without a customer stay `unmatched`.

### 6.2 Manual / CSV POSTs (`app/api/admin/commissions/sales/route.ts`)
`matchSaleIdentity()` applies the priority documented in the principle list:

1. **HOT mobile id** — look up in `customer_hot_accounts` (pending/active/inactive). Single match = `matched` with confidence `1.0` and resolved `customer_hot_account_id`. Multiple matches = `ambiguous`.
2. **Phone** — normalised (strip spaces, dashes, dots) and matched against `customers.phone`. Single = `matched` with confidence `0.9`. Multiple = `ambiguous`.
3. **Fallback** — `unmatched`, no IDs written.

All three outcomes preserve `hot_mobile_id_snapshot` from the request so the original import data is never lost, even when there is no match.

## 7. Architectural decisions

These are recorded so the next engineer does not relitigate them.

| Decision | Why |
|---|---|
| **DB sequence + trigger for `customer_code`** | Removes app-side retry loops and makes collisions structurally impossible. |
| **Partial unique index on active HOT IDs** | Lets a HOT mobile be transferred between customers via close + recreate without blocking on prior cancelled rows. |
| **Soft-cancel for HOT accounts** | Keeps the historical record intact — required for commission reconciliation and audits. |
| **Transfer via close + recreate (not UPDATE customer_id)** | An UPDATE would erase the original owner and break the audit trail. Close + recreate is reversible and observable. |
| **`id_number` never auto-filled, never in public responses** | Security over convenience. The customer re-typing it is an acceptable friction in Phase 1. Auto-fill can come later via a dedicated, rate-limited endpoint. |
| **Returning-customer flow uses existing `/store/auth?return=`** | The OTP infrastructure, rate limiting, and token storage already exist. Reinventing them in the cart would duplicate logic and diverge security models. |
| **Reuse `localStorage.clal_customer_token`** | `/store/auth` writes it and `/store/account` reads it; switching to sessionStorage silently breaks `/store/account`. Moving to secure HttpOnly cookies is a cross-cutting concern tracked for Phase 2/3. |
| **CRM edits in existing modal, not a new `[id]/page.tsx`** | The list-page modal is a working pattern with no missing affordances. A separate route page is a UX decision that can be taken later. |
| **Audit via the existing `audit_log` table** | No new helper, no new table. The schema already supports `entity_type / entity_id / details`. |
| **Client-side match filter on commissions history** | The admin page already fetches a bounded list; adding a server-side filter would require new query params on a stable route. Client filter keeps PR6 additive. |

## 8. Lifecycle examples

### 8.1 Brand-new customer checkout
1. Cart collects name, phone, address.
2. `POST /api/orders` hits the `INSERT` branch, trigger issues `customer_code`.
3. Response returns `{ customerCode: "CLAL-100123", isNewCustomer: true }`.
4. WhatsApp template and HTML email both render the code block.
5. Confirm step renders the brand card with "احتفظ به".

### 8.2 Returning customer with existing code
1. Customer opens cart, sees banner, taps "استرجاع بياناتي".
2. Redirected to `/store/auth?return=/store/cart`, completes OTP.
3. Auth route writes `clal_customer_token` in localStorage (unchanged code path).
4. Cart mount effect sees the token, calls `GET /api/customer/profile`, prefills fields.
5. `POST /api/orders` hits the `UPDATE` branch; `customer_code` is already set, no backfill.
6. Response returns `{ customerCode, isNewCustomer: false }`. Notifications suppress the "new code" card.

### 8.3 Legacy customer (`customer_code IS NULL`)
1. Customer places an order via the store.
2. Order route runs the `UPDATE` branch, detects null `customer_code`, and calls `assign_code_to_existing_customer(id)`.
3. RPC issues a code via the same sequence, returns it to the route.
4. From this moment on the customer looks identical to a fresh one — their `isNewCustomer` flag is still `false` (they're not new), but `customerCode` is populated. Notifications do not show the new-code card.

### 8.4 HOT transfer between two customers
1. Admin opens the *old* customer in CRM, clicks the 🗑️ icon on the relevant HOT account. Backend sets `status='cancelled'`, `is_primary=false`, `ended_at=now()`. Audit log records `hot_account.cancel`.
2. Admin opens the *new* customer, uses "+ إضافة" to create a new `customer_hot_accounts` row with the same `hot_mobile_id`. The partial unique index allows it because the previous row is no longer active. Audit log records `hot_account.create`.
3. Commission sync from that point onward maps orders to the new owner automatically.

### 8.5 Unmatched CSV import
1. Admin uploads a CSV of manual sales via `POST /api/admin/commissions/sales`.
2. `matchSaleIdentity()` cannot locate the customer (unknown phone, no HOT mobile id).
3. Row is persisted with `match_status='unmatched'`, `customer_id=null`, `hot_mobile_id_snapshot` preserving whatever the CSV contained.
4. Admin opens `/admin/commissions/history`, filters by "unmatched", and reviews rows for manual reconciliation.

## 9. Runbook

### 9.1 Adding a HOT account to a customer
1. Open `/crm/customers`, search for the customer by name / phone / `customer_code` / `hot_mobile_id`.
2. Click the customer row to open the modal.
3. Under "📱 أكواد HOT", click "+ إضافة".
4. Enter `hot_mobile_id` (required). Optional: label, is_primary.
5. Save. If the HOT ID is already active elsewhere, the backend returns 409 — resolve by transferring first.

### 9.2 Transferring a HOT account between customers
1. On the current owner, soft-cancel the HOT account (🗑️).
2. On the new owner, add a new HOT account with the same `hot_mobile_id`.
3. Both actions are logged in `audit_log` with the admin user.
4. Reconcile commission history if any prior sales were attributed to the old owner (manual task in Phase 1).

### 9.3 Resolving an "ambiguous" commission row
1. Open `/admin/commissions/history`, filter by "ambiguous".
2. The badge tooltip (when available) shows the customer code snapshot.
3. Use the phone / HOT number from the sale and manually pick the correct customer in CRM.
4. Edit the sale row via the PUT endpoint with `customer_id` set explicitly (use the existing edit UI once it's wired, or an admin SQL for Phase 1).

### 9.4 Backfilling `customer_code` for legacy customers
- **Trickle-in (default):** happens automatically on the next order — no action needed.
- **One-shot:** run the commented-out SQL in `20260411000001_customer_retention_phase1.sql` — `UPDATE customers SET customer_code = 'CLAL-' || LPAD(nextval('customer_code_seq')::text, 6, '0') WHERE customer_code IS NULL;`. Sequence ensures no collisions.

### 9.5 Changing a customer's phone (Phase 1 workflow)
Not self-service. The cart form does not allow editing phone, and neither does the CRM modal. An admin may run a one-off SQL update after verifying identity through another channel. Phase 2 will add a proper audited workflow.

## 13. Testing notes (manual regression)

Run these after each migration is applied to staging:

1. `SELECT nextval('customer_code_seq');` returns a value `>= 100000`.
2. Insert a customer via the store; confirm `customer_code` matches `CLAL-\d{6}`.
3. `UPDATE customers SET customer_code='HACK' WHERE id=...;` raises the protection exception.
4. Insert a customer with phone `050-111-2222`; select the row and confirm phone is stored as `0501112222`.
5. Hit `POST /api/auth/customer` with `verify_otp` and confirm `customer.customer_code` is in the response (and `id_number` is absent).
6. Hit `GET /api/customer/profile` with the token from step 5; confirm `customer_code` is present.
7. Place a full store order as a new customer; confirm notifications show the code and the confirm step renders the brand card.
8. Returning flow: log out, add items, tap the CTA, finish OTP, verify the cart prefills everything except `id_number`.
9. CRM: add a HOT account; repeat with a duplicate `hot_mobile_id` on a second customer and confirm 409.
10. CRM: soft-cancel the duplicate on customer A; add the same HOT to customer B; expect success.
11. CRM search: `CLAL-XXXXXX` returns the matching customer; `?hot_mobile_id=...` returns the correct customer.
12. Run the commissions auto-sync; confirm new rows have `customer_id`, `match_status='matched'`, `match_method='order_sync'`.
13. CSV-style manual POST to `/api/admin/commissions/sales`: one row with a known HOT, one with an unknown phone. Verify `match_status` values and `hot_mobile_id_snapshot` preserved.
14. Commissions history page: filter by "unmatched" and confirm only `match_status='unmatched'` rows remain; hover a row to see the tooltip with the customer code snapshot.

### Security spot-checks
- `grep -r "id_number" app/api/customer` — no public GET response should reference it.
- `/api/crm/customers/[id]` without a valid admin session returns 401.
- `POST /api/crm/customers/[id]/hot-accounts` with a non-admin token returns 401/403.
- Anon or authenticated (non-service) selects against `customer_hot_accounts` return empty because of RLS.
