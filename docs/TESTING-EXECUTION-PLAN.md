# خطة الفحص التنفيذي المرجعية

> هذا الملف هو المرجع التنفيذي الأعلى للفحص في مشروع `ClalMobile`.  
> الغاية منه ليست وصف "أنواع الاختبارات" فقط، بل ضبط **متى يحق لنا القول إن شيئًا ما فُحص فعلًا**، وكيف ننفذ فحوصًا حيّة آمنة، وكيف ننظف فقط البيانات التي أنشأها الفحص.

## المبدأ الحاكم

لا يجوز القول:

- `تم فحصه`
- `كل شيء يعمل`
- `جاهز للإطلاق`
- `مغلق`

إلا إذا كان ذلك **مدعومًا بأثر تنفيذي حقيقي**:

- أمر تم تشغيله
- تقرير أو سجل
- لقطة متصفح أو نتيجة API
- أو `manifest` موثق لجولة الفحص

إذا لم يحدث ذلك، فالتعبير الصحيح يكون واحدًا من الآتي فقط:

- `مغطى باختبار مكتوب لكنه لم يُشغّل في هذه الجولة`
- `مفحوص محليًا فقط`
- `مفحوص على المتصفح محليًا فقط`
- `مفحوص حيًا على الإنتاج في وضع آمن`
- `غير مفحوص بعد`
- `مقيّد ويحتاج خطة استرجاع قبل الفحص الحي`

## قاموس الصراحة الإلزامي

كل تقرير فحص أو تسليم يجب أن يستخدم الحالات التالية حرفيًا:

| الحالة | معناها |
|---|---|
| `مغطى بالكود` | يوجد اختبار أو سكربت، لكن لم يُشغّل ضمن الجولة الحالية |
| `مفحوص محليًا` | تم تشغيل الفحص على البيئة المحلية ونجح |
| `مفحوص بالمتصفح` | تم تنفيذ التفاعل فعليًا عبر المتصفح ورُوجع سلوكه |
| `مفحوص حيًا` | تم على البيئة الحية أو نسخة منشورة مقابلة لها ضمن قواعد الأمان |
| `مقيّد` | لا يجوز فحصه حيًا بعد بسبب غياب خطة استرجاع أو لوجود أثر تدميري |
| `فشل` | تم تشغيله وفشل، مع وجود أثر واضح للفشل |
| `غير مفحوص` | لم يُختبر بعد في هذه الجولة |

## ما الذي نعالجه بهذه الخطة

هذه الخطة تعالج الفجوة بين:

- نجاح `build`
- وجود اختبارات متفرقة
- وبين إثبات أن:
  كل زر
  كل حقل
  كل رفع ملفات
  كل حفظ
  كل تكامل
  كل Redirect
  كل رسالة نجاح/فشل
  كل استدعاء شبكة
  وكل أثر جانبي

قد تم التحقق منه طبقيًا وبأثر قابل للمراجعة.

## طبقات الفحص المعتمدة

### الطبقة 0 — الثبات الأساسي

تشمل:

- `TypeScript`
- `Lint`
- `Build`
- فحص المتغيرات الحرجة
- التأكد من أن الأوامر الأساسية لا تنهار

أمثلة:

- `npm run build`
- `npm run test:run`

### الطبقة 1 — منطق ودوال وعقود API

تشمل:

- وحدات المنطق
- عقود API
- الحماية
- التحقق من الصلاحيات
- التحقق من CSRF
- التحقق من الرسائل والبنى المعادة

### الطبقة 2 — فحص واجهة محلي تفاعلي

تشمل:

- النقر
- الإدخال
- الحفظ
- الرفع
- النوافذ
- الرسائل
- الانتقال بين الصفحات
- نظافة الـ console
- حالات الشبكة الفاشلة

### الطبقة 3 — فحص منشور غير حي إن وجد

تشمل:

- نسخة منشورة قبل الإنتاج أو بيئة آمنة مقابلة
- نفس تدفقات الطبقة 2 لكن على بنية قريبة من الواقع

### الطبقة 4 — فحص حي آمن `live-safe`

تشمل:

- التدفقات المسموح تنفيذها حيًا من دون الإضرار ببيانات حقيقية
- أو التي نملك لها خطة استرجاع واضحة

وهذه الطبقة لا تعمل إلا إذا توفر:

- `TEST_RUN_ID`
- `TEST_MANIFEST_PATH`
- بيانات اختبار صريحة
- وخطة تنظيف بعد التنفيذ

### الطبقة 5 — التكاملات والـ webhooks والآثار غير المتزامنة

تشمل:

- البريد
- الواتساب
- الدفع
- الـ webhooks
- الذكاء الاصطناعي
- التخزين
- الإشعارات

### الطبقة 6 — التقرير التنفيذي والتنظيف

تشمل:

- تقرير ما الذي فُحص فعلًا
- ما الذي لم يُفحص
- ما الذي كان مقيّدًا
- وما الذي أُنشيء من بيانات أو ملفات
- ثم حذف أو استرجاع **فقط** أثر جولة الاختبار

## قواعد الفحص الحي

الفحص الحي ليس "تشغيلًا حرًا" على الإنتاج. القواعد:

1. لا إرسال لعملاء حقيقيين.
2. لا دفع حقيقي.
3. لا OTP حقيقي لرقم غير مخصص للاختبار.
4. لا تعديل دائم على بيانات تشغيلية من دون خطة استرجاع.
5. كل جولة حيّة يجب أن تحمل `runId`.
6. كل أثر يجب أن يسجل في `manifest`.
7. كل جولة يجب أن يعقبها `cleanup`.

## الـ Manifest

الـ `manifest` هو سجل التنفيذ الحي أو شبه الحي لجولة الاختبار.

يجب أن يحتوي على:

- `runId`
- البيئة
- الرابط المستهدف
- اسم المجموعة أو السيناريو
- وقت البداية
- قائمة الآثار المسجلة
- نتائج التنظيف

أمثلة على الآثار التي تُسجّل فيه:

- صف أُنشئ في جدول
- صف يجب استرجاعه لقيمته السابقة
- ملف رُفع إلى التخزين
- إعداد تغيّر ويجب إرجاعه
- ملاحظة تشغيلية أو قيد معروف

## أنواع الآثار الأولية المعتمدة

النسخة الأولى من أدوات الخطة تدعم:

- `row_upsert_restore`
- `row_update_restore`
- `db_delete_eq`
- `db_delete_like`
- `storage_remove_path`
- `storage_remove_public_url`
- `note`

## قاعدة التنظيف

التنظيف لا يحذف "كل ما يشبه بيانات الاختبار".  
التنظيف يحذف أو يسترجع **فقط** ما تم تسجيله في `manifest` الخاص بنفس الجولة.

هذا يمنع:

- حذف بيانات لا تخص الجولة
- العبث بآثار فرق أخرى
- ضياع أثر مطلوب للمراجعة

## مسار التشغيل المعتمد

### 1. تجهيز الجولة

```bash
npm run test:live:prepare
```

المخرجات:

- `TEST_RUN_ID`
- `TEST_MANIFEST_PATH`
- ملف manifest فارغ/جاهز

### 2. تشغيل الفحوص

يتم تشغيل الفحوص المحلية/المتصفحية/الحيّة المناسبة، مع تسجيل الأثر في `manifest`.

### 3. المراجعة

كل تدفق يجب أن يسجل نتيجته بأحد الأوصاف المعتمدة:

- `مفحوص محليًا`
- `مفحوص بالمتصفح`
- `مفحوص حيًا`
- `فشل`
- `غير مفحوص`
- `مقيّد`

### 4. التنظيف

```bash
npm run test:live:cleanup
```

أو:

```bash
npm run test:live:cleanup -- --manifest .tmp/live-test-runs/<run-id>.json
```

ولفحص السكربت نفسه من دون لمس القاعدة:

```bash
npm run test:live:cleanup -- --manifest .tmp/live-test-runs/<run-id>.json --dry-run
```

## ما الذي يعد مقيّدًا الآن

حتى لحظة كتابة هذا الملف، لا يجوز وصف العناصر التالية بأنها "مفحوصة حيًا" إلا بعد استكمال آلية استرجاعها:

### 1. رفع شعار المتجر إذا كان هناك شعار قائم

السبب:

- المسار الحالي لرفع الشعار يحذف الشعار السابق من التخزين قبل تثبيت الجديد
- لذلك لا يكفي استرجاع `logo_url` في الإعدادات فقط
- يلزم قبل الفحص الحي:
  - إما أن تكون البيئة بلا شعار قائم أصلًا
  - أو أن نضيف آلية Snapshot/Restore كاملة للملف السابق

لذلك حالته الحالية:

- `مفحوص محليًا` بعد إصلاح CSRF
- `مقيّد` للفحص الحي الكامل إذا كان هناك شعار قائم نحتاج استرجاعه

### 2. الدفع الحقيقي

حالته:

- `مقيّد`

إلا في حالة وجود بيئة Sandbox أو حسابات اختبار مخصصة.

### 3. رسائل العملاء الفعلية

حالته:

- `مقيّد`

إلا عبر أرقام/بريد/قنوات اختبار مخصصة.

## تقرير الجولة الإلزامي

كل جولة تنفيذ يجب أن تنتهي بجدول مختصر مثل هذا:

| العنصر | الحالة | البيئة | الدليل | ملاحظات |
|---|---|---|---|---|
| تسجيل الدخول | مفحوص محليًا | local | أمر + متصفح | ناجح |
| استعادة كلمة المرور | مفحوص محليًا | local | أمر + متصفح | ناجح |
| رفع الشعار | مفحوص محليًا | local | متصفح + شبكة | أصلح CSRF |
| رفع الشعار حيًا | مقيّد | production | — | يحتاج استرجاع ملف الشعار السابق |

## المرحلة التنفيذية الحالية

هذه الدفعة لا تدّعي أنها أنجزت "الفحص الشامل" كله.  
الذي تنجزه فعليًا هو:

1. تثبيت مرجع الصراحة والتنفيذ
2. إنشاء أدوات `prepare` و`manifest` و`cleanup`
3. إصلاح عطل رفع الشعار الحالي
4. بدء تحويل الفحص من وعود عامة إلى تشغيل منضبط قابل للتتبع

## قاعدة أخلاقية تنفيذية

إذا لم يُشغّل شيء، فلن يُقال إنه شُغّل.  
إذا لم يُفحص شيء، فلن يُقال إنه فُحص.  
إذا كان الفحص مقيّدًا، فسيُقال إنه مقيّد بوضوح، لا أن يُمرر على أنه "غالبًا يعمل".
